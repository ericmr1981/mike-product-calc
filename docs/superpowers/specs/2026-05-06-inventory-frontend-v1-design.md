# 门店优先库存前端改造设计（V1）

- 日期：2026-05-06
- 状态：Draft（已与用户确认方向：门店优先 + 双端自适应）
- 目标版本：V1（先上线可用，再做视觉与交互增强）

## 1. 背景与目标

当前系统已具备库存快照上传与 Supabase 存储能力，但前端侧缺少“门店日常可直接使用”的库存驾驶舱。V1 的目标是：

1. 让门店在手机与电脑上都能快速看到库存风险。
2. 提供可执行的补货优先级，减少断货。
3. 给总部/老板保留后续扩展入口，但不扩大首期范围。

## 2. 方案对比与选择

### 方案 A：纯 Streamlit 改造（选中）
- 做法：保留现有 Streamlit 架构，在现有应用内新增门店库存页面与组件层。
- 优点：上线快、改动风险低、复用现有 Supabase 与 Python 逻辑。
- 缺点：移动端交互精细度上限有限。

### 方案 B：Streamlit + 嵌入式定制组件
- 做法：主体 Streamlit，关键卡片/交互使用 components 定制。
- 优点：视觉与交互上限更高。
- 缺点：工程复杂度与维护成本上升。

### 方案 C：独立 Next.js 前端
- 做法：新建 Web 前端，Supabase 为核心数据层。
- 优点：移动端体验最佳，长期扩展性强。
- 缺点：首期周期长，不符合“先快速上线门店”。

**V1 决策：方案 A。**
**V1 后演进：视门店反馈升级到方案 B。**

## 3. 用户与场景

### 3.1 主要用户
- 门店店长/店员（V1 主目标）

### 3.2 次要用户（V1 仅兼容，不主打）
- 总部运营/采购
- 老板/管理层

### 3.3 核心任务
1. 开店前快速确认是否有缺货风险。
2. 营业中查看低库存并安排补货。
3. 发现异常数据（负库存、金额异常）并上报处理。

## 4. 页面信息架构（V1）

1. 门店总览（Dashboard）
- KPI：可用品项数、低库存数、缺货数、异常数。
- 快照时效：最近快照时间 + 超时提醒。
- 今日重点：Top N 高风险品项。

2. 库存列表（List）
- 筛选：仓库、品类、状态（正常/低库存/缺货/异常）、关键词。
- 列：品项编码、品项名称、可用量、库存量、单位、状态、更新时间。
- 排序：默认按风险优先级降序。

3. 补货建议（Replenishment）
- 规则：按缺货优先、低库存其次、可用量缺口排序。
- 输出：建议补货品项、建议量、紧急等级。

4. 异常中心（Anomalies）
- 负库存项。
- 金额不一致项（库存量 * 单价 与现存金额偏差超阈值）。
- 规格缺失项（告警级）。

## 5. 双端体验策略

### 5.1 手机端
- 顶部 4 张 KPI 卡片（横向可滑动）。
- 底部 Tab 导航：总览 / 列表 / 补货 / 异常。
- 行项采用大点击区域，减少误触。

### 5.2 桌面端
- 左侧筛选栏 + 中间列表 + 右侧详情面板。
- 支持更丰富列信息与批量查看。

### 5.3 共用原则
- 同一业务规则，同一数据源，不做双套计算逻辑。
- 仅布局与信息密度因断点变化。

## 6. 数据与接口设计

### 6.1 数据来源
- 表：`inventory_snapshot_batches`、`inventory_snapshot_items`
- 视图：`v_inventory_latest_item_by_warehouse`

### 6.2 前端读取口径
- 默认读取“每仓每品项最新快照”。
- 展示最近快照时间（来自 batch 的 `snapshot_at`）。

### 6.3 状态判定规则（V1）
- 缺货：`available_qty <= 0`
- 低库存：`available_qty > 0 且 <= reorder_point`
  - V1 固定默认：`reorder_point = 5`（前端可调参数，默认值为 5）
  - 后续版本：切换为门店级阈值配置表
- 异常：`is_negative_stock = true` 或 `has_amount_mismatch = true`

## 7. 错误处理与可观测性

1. 数据不可用
- 提示“暂无快照数据”，并给出最近一次成功导入时间（若存在）。

2. 快照过期
- 若当前时间与 `snapshot_at` 超过阈值（例如 2 小时），页面高亮“数据可能过期”。

3. 查询失败
- 显示可读错误卡片，保留重试按钮。

4. 观测指标（V1）
- 页面加载耗时。
- 快照时效。
- 异常项数量趋势。

## 8. 测试策略

1. 规则测试
- 缺货/低库存/异常判定单元测试。

2. 数据回归
- 用已分析样本（182 行）验证统计一致性。

3. 双端验收
- 手机断点与桌面断点下布局与核心交互检查。

4. 容错验证
- 无数据、过期数据、字段缺失场景。

## 9. 实施范围与里程碑

### M1（V1 核心）
- 新增门店库存驾驶舱页面（总览/列表/补货/异常）。
- 接入最新快照读取与状态判定。
- 完成双端基础响应式布局。

### M2（V1.1 体验增强）
- 优化视觉层级与交互反馈。
- 增加筛选预设与高频快捷操作。

### Out of Scope（本轮不做）
- 独立 Next.js 前端。
- 复杂权限体系。
- 自动调拨策略引擎。

## 10. 风险与缓解

1. 风险：快照上传不及时导致门店误判库存。
- 缓解：显著展示快照时间与过期提示。

2. 风险：门店阈值定义不统一导致“低库存”噪声。
- 缓解：V1 先提供默认阈值，后续支持门店级阈值配置。

3. 风险：单页逻辑继续膨胀。
- 缓解：将库存视图逻辑抽离为独立函数模块，避免 `app.py` 继续耦合。

# 概览页驾驶舱 UI 重设计（V1）

- 日期：2026-05-06
- 状态：Draft（方向已确认：管理驾驶舱风 + 风险在上经营在下）
- 目标页面：Tab1「概览/校验」

## 1. 背景

当前 Tab1 主要是基础指标平铺，能看数据但不够“决策导向”。
用户希望概览页更像驾驶舱：第一眼看到风险，第二眼看到经营健康，并知道下一步应该去哪一页处理问题。

## 2. 设计目标

1. 风险优先：缺货、异常、库存快照时效放在首屏最醒目区域。
2. 经营次级：原料/产品/规格等经营指标作为第二层信息。
3. 行动导向：给出“下一步去哪里处理”的明确引导。
4. 双端友好：桌面与移动端都能快速读取核心状态。

## 3. 方案选择

### 方案 A（轻改）
- 仅改颜色和排列，不改信息结构。
- 优点：最快。
- 缺点：改观有限。

### 方案 B（选中，中改驾驶舱）
- 重构 Tab1 信息架构为“两层仪表盘 + 动作建议”。
- 优点：提升可读性与决策效率，开发成本可控。
- 缺点：需要新增聚合逻辑与样式层。

### 方案 C（重构首页框架）
- 抽离通用组件，做全局控制塔。
- 优点：长期最优。
- 缺点：范围过大，不适合当前迭代。

**决策：采用方案 B。**

## 4. 信息架构（Tab1）

## 4.1 顶部状态条
- 标题：运营控制台概览
- 数据时间：最后缓存加载时间
- 快捷动作：刷新缓存按钮（保留）

## 4.2 第一层：风险区（优先）
卡片（高对比配色，红/橙）：
1. 缺货项数量
2. 异常项数量
3. 库存快照过期状态

呈现规则：
- 若风险 > 0，卡片高亮并显示状态标签（高/中/低）
- 若风险为 0，显示“正常”样式

## 4.3 第二层：经营区（次优先）
卡片（冷色系，蓝/青）：
1. 原料总数
2. 产品总数
3. 最终成品数
4. 出品规格数
5. 已上线原料数（可并入原料卡副信息）

## 4.4 第三层：建议动作区
- 显示 3~4 条建议动作（按当前风险动态排序）
- 示例：
  - 有缺货 → 去 Tab8 查看库存缺口
  - 有补货需求 → 去 Tab4 生成补货计划
  - 有规格覆盖问题 → 去 Tab7 校验出品规格

## 5. 数据来源与口径

## 5.1 经营指标
- 来源：现有 `st.session_state.cached_*`

## 5.2 风险指标
- 缺货项/异常项：来自 `v_inventory_latest_item_by_warehouse` 聚合
- 快照过期：来自最新 `snapshot_at` 与当前时间差（阈值 2 小时）

## 5.3 口径说明
- 风险区所有指标需标注“基于最新库存快照”
- 当库存数据不可用时，风险区显示“数据未就绪”而不是空值

## 6. 交互与视觉规范

1. 卡片优先级
- 风险卡高度略大于经营卡
- 风险卡使用状态色块 + 图标（文本符号）

2. 文案策略
- 每张卡只保留：主数字 + 一句辅助文案
- 避免重复标题（减少视觉噪音）

3. 响应式策略
- 桌面：风险区 3 卡横排，经营区 4 卡横排
- 移动：风险区纵向堆叠置顶，经营区紧随

## 7. 边界与容错

1. 无库存快照
- 风险卡显示“未同步库存”，并给出操作提示

2. Supabase 接口失败
- 显示可读错误卡，不影响经营区显示

3. 缓存未加载
- 显示 skeleton/占位文案，避免白屏感

## 8. 测试策略

1. 组件级
- 风险卡颜色/文案在不同状态下切换正确

2. 数据级
- 风险指标与库存页面汇总一致
- 快照过期阈值判断正确

3. 回归
- 不影响 Tab1 现有刷新逻辑
- 不影响其他 Tab 加载

## 9. 范围约束

### 本次包含
- Tab1 信息架构重排
- 风险区 + 经营区 + 动作建议区
- 基础响应式优化

### 本次不包含
- 全站主题重构
- 新图表库引入
- 全局组件系统抽离

## 10. 风险与缓解

1. 风险：风险数据查询增加首屏负担
- 缓解：复用缓存与轻量聚合，失败时降级展示

2. 风险：样式改动影响现有 Tab1 可读性
- 缓解：保留旧指标名称与口径，仅改布局与层级

3. 风险：视觉风格与现有页面不一致
- 缓解：沿用当前配色变量体系，在其上做优先级强化
